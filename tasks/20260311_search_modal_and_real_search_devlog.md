# 2026-03-11 搜索弹窗交互与真实搜索修复开发文档

## 一、需求背景

本次开发来自两个直接的用户反馈：

1. 项目侧边栏的搜索功能在“未输入任何内容”时，看不到关闭按钮，用户不知道如何关闭搜索弹窗
2. 当前搜索功能看起来不像真正完成了“可用搜索”，因为随便输入一些字符经常搜不到结果，用户希望确认这到底是数据问题、前端问题，还是搜索本身没有打通

这两个问题表面上分别属于：

- 搜索弹窗交互问题
- 搜索能力真实性问题

但在实际排查后发现，它们分别涉及前端显示层、前后端接口对接、后端查询实现三条链路，因此需要一并修复。

---

## 二、问题现状

### 2.1 搜索弹窗关闭交互问题

全局搜索弹窗的实现位于：

- `client/src/components/Search/SearchModal.js`

原始行为是：

- 输入框右侧只有在 `query` 非空时，才显示一个 `X`
- 这个 `X` 的真实作用是“清空输入”，不是“关闭弹窗”
- 用户未输入内容时，这个按钮根本不会出现

理论上，用户似乎可以点击搜索区域外部关闭弹窗，但实际 DOM 结构里：

- 外层容器绑定了“点击空白关闭”
- 遮罩层是单独一层 `motion.div`，覆盖了整屏
- 点击黑色遮罩时，并不会命中外层容器上的那段关闭逻辑

最终用户体验就是：

- 输入为空时，顶部没有显式关闭按钮
- 点击外围黑色遮罩也不稳定关闭
- 用户会觉得搜索弹窗“没法关”

### 2.2 搜索结果真实性问题

用户反馈“随机搜字符都失败”，我先按链路拆开分析：

#### 前端调用链

搜索弹窗会并行调用两个接口：

- `searchSref(query, 5)` -> `/api/sref?search=...`
- `searchGallery(query, 5)` -> `/api/gallery?search=...`

对应实现位于：

- `client/src/services/searchApi.js`

#### 前端结果解析问题

搜索弹窗中，`sref` 结果读取代码原本是：

```js
setSrefResults(srefRes?.data?.srefs || srefRes?.data?.data || []);
```

但 `sref` 接口实际返回的是：

```json
{
  "posts": [...],
  "pagination": {...}
}
```

这意味着：

- 前端始终没有正确读取 `sref` 搜索结果
- `explore` 搜索这一路看起来像“总是失败”
- 即使后端查到了内容，前端也会把结果当成空数组

这是一个明确的前后端字段对接错误。

#### Gallery 搜索实现问题

`gallery` 路由的搜索逻辑原先主要依赖 Mongo `$text`：

- 列表接口 `GET /api/gallery` 中，如果有 `search`，会构造 `$text: { $search: search }`
- 专门的 `/api/gallery/search` 也是 `$text` 搜索

这种实现的特点是：

- 对全文索引友好
- 对“像搜索引擎一样的自然关键词”有一定帮助
- 但对短字符串、单字符、随机字符并不友好

用户视角里，搜索输入往往会包含：

- 单个字母
- 短词
- 试探性的随机字符
- 含特殊符号的字符串

而 `$text` 对这类输入常常返回 0 条，哪怕实际数据中明明有很多可匹配内容。

#### Sref 搜索正则安全问题

`sref` 路由原本使用：

```js
const re = new RegExp(search.trim(), 'i');
```

这意味着如果用户输入：

- `?`
- `(`
- `*`
- `+`

这类正则元字符，就可能导致：

- 正则报错
- 搜索逻辑异常
- 前端感知为“搜索失败”

所以这不是单纯“没搜到”，而是“实现上不安全”。

---

## 三、根因总结

本次搜索问题并不是单一 bug，而是三层问题叠加：

### 根因 1：关闭按钮语义混淆

- 输入框右侧唯一的 `X` 是“清空输入”
- 它只在有输入时出现
- 用户没有一个稳定的“关闭搜索”入口

### 根因 2：Sref 搜索结果字段接错

- 后端返回 `posts`
- 前端读的是 `srefs`
- 导致 `explore` 搜索长期空结果

### 根因 3：Gallery 搜索实现太严格

- 使用 `$text` 搜索
- 对单字符、短词、随机字符和很多用户试探性输入不友好
- 用户主观上会认为“搜索坏了”

### 根因 4：Sref 搜索没做正则转义

- 特殊字符会被当成正则元字符
- 某些输入会直接破坏查询

---

## 四、改造目标

这次改造的目标不是只修一处显示问题，而是让搜索在“用户使用感知上”真正可用：

1. 搜索弹窗在任何输入状态下都有明确关闭方式
2. 点击搜索区域外的遮罩层可以关闭弹窗
3. `explore` 搜索结果要真实显示出来
4. `gallery` 搜索要支持更接近用户直觉的模糊搜索
5. 搜索输入中的特殊字符不能导致异常

---

## 五、具体实现

### 5.1 搜索弹窗交互修复

文件：

- `client/src/components/Search/SearchModal.js`

改动点：

#### 改动 A：让遮罩层直接负责关闭

之前点击外围不稳定关闭，是因为关闭逻辑绑在外层容器，而遮罩层自己没有 `onClick`。

现在改成：

- 遮罩层 `motion.div` 直接绑定 `onClick={onClose}`

这样用户点击黑色背景时，会稳定关闭弹窗。

#### 改动 B：新增常驻关闭按钮

在输入框右侧增加一个专门的关闭按钮：

- 始终可见
- 语义是关闭搜索弹窗
- 与“清空输入”的按钮分开

这样即使输入为空，用户也能直接关闭搜索弹窗。

#### 改动 C：保留清空输入按钮

原本只有输入非空时出现的 `X` 现在仍然保留，但它只负责：

- 清空当前输入内容

用户因此拥有两个不同语义的操作：

- `清空输入`
- `关闭弹窗`

这比以前更清晰。

---

### 5.2 修复 SearchModal 对 sref 结果的读取

文件：

- `client/src/components/Search/SearchModal.js`

原本代码：

```js
setSrefResults(srefRes?.data?.srefs || srefRes?.data?.data || []);
```

修复后改为：

```js
setSrefResults(srefRes?.data?.posts || srefRes?.data?.srefs || srefRes?.data?.data || []);
```

修复效果：

- 现在能正确读取 `sref` 接口返回的 `posts`
- `explore` 搜索结果不再被前端误判为空

---

### 5.3 修复 SearchModal 的 sref 展示字段

文件：

- `client/src/components/Search/SearchModal.js`

原来的展示逻辑里，`sref` 列表使用：

- `item.images?.[0]`
- `item.sref || item.code || item._id`

但现有接口返回结构更稳定的字段是：

- `previewImage`
- `srefCode`

因此现在改成优先使用：

- `item.previewImage`
- `item.srefCode`

这样弹窗里的 `sref` 搜索结果展示更准确，也与项目其他页面的数据结构一致。

---

### 5.4 把 gallery 搜索从严格全文搜索改为安全模糊匹配

文件：

- `server/routes/gallery.js`

新增工具函数：

```js
const escapeRegex = (value = '') => value.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
```

作用：

- 把用户输入中的正则元字符转义
- 避免构造 `RegExp` 时出错

#### 改动 A：列表接口 `GET /api/gallery`

当存在 `search` 时，不再使用 `$text`，而改成：

- `title`
- `prompt`
- `description`
- `tags`
- `sourceAuthor`

上的正则模糊匹配。

这会让搜索行为更贴近用户直觉：

- 输入短词可匹配
- 输入单字符可匹配
- 输入特殊字符不会炸掉

#### 改动 B：专用接口 `GET /api/gallery/search`

这个接口也同步改成正则模糊匹配，并允许只要输入非空就能搜索，而不是强制长度至少 2。

排序策略也从 `$textScore` 改为：

- `views: -1`
- `createdAt: -1`

因为改成正则搜索后，已经没有 `$textScore` 可用。

---

### 5.5 修复 sref 搜索的正则安全问题

文件：

- `server/routes/sref.js`

新增同样的：

```js
const escapeRegex = ...
```

并把：

```js
new RegExp(search.trim(), 'i')
```

改成：

```js
new RegExp(escapeRegex(search.trim()), 'i')
```

这样：

- `?`
- `(`
- `*`
- `+`

等特殊字符不再导致搜索异常。

## 六、验证过程

### 6.1 前端构建验证

执行：

```bash
cd client
npm run build
```

结果：

- 构建成功
- 没有新增阻断性错误
- 仓库里仍存在一些历史 ESLint warning，但与本次搜索修复无直接关系

### 6.2 数据层验证：证明旧搜索为什么“像坏的一样”

我使用 Node + Mongo 直接对比了旧式全文搜索和模糊匹配的结果差异。

以 `gallery` 数据为例：

- 搜索词 `a`
  - `$text` 命中：`0`
  - 正则模糊匹配：`429`

- 搜索词 `ab`
  - `$text` 命中：`1`
  - 正则模糊匹配：`203`

- 搜索词 `art`
  - `$text` 命中：`31`
  - 正则模糊匹配：`197`

- 搜索词 `x`
  - `$text` 命中：`2`
  - 正则模糊匹配：`346`

这说明用户说“随机输字符都失败”并不是错觉，而是旧的 `$text` 搜索模型本身就和用户预期不一致。

### 6.3 修复后验证

修复后的服务器侧统计验证如下：

- `a`
  - `galleryCount = 429`
  - `srefCount = 1306`

- `ab`
  - `galleryCount = 205`
  - `srefCount = 1`

- `art`
  - `galleryCount = 199`
  - `srefCount = 1190`

- `?`
  - `galleryCount = 2`
  - `srefCount = 0`
  - 关键点：不会报错

这说明：

- 搜索已经具备真实结果返回能力
- 特殊字符不再破坏搜索流程

## 九、后续建议

虽然这次已经把搜索修到“真实可用”，但如果后续还想进一步提升，可以考虑：

1. 给搜索结果增加高亮匹配片段
2. 增加最小输入长度提示，而不是默默空结果
3. 对搜索做分类 tabs，例如 `All / Explore / Gallery`
4. 把 `seedance` 也纳入全局搜索
5. 为搜索建立更明确的排序策略，例如：
   - 精确匹配优先
   - 标题命中优先于正文命中
   - 新内容与热门内容平衡排序

## 十、结论

本次更新不是简单“加一个关闭按钮”，而是完整修复了搜索功能的两个核心问题：

- 搜索弹窗关闭交互不清晰
- 搜索结果链路不完整且搜索实现不够贴近用户预期

经过本次改造后：

- 搜索弹窗可关闭性明显改善
- `sref` 与 `gallery` 搜索都进入真实可用状态
- 对短词、随机字符和特殊字符的处理明显更稳
