# 阶段13 - Modal 关闭后滚动位置异常完整修复日志

**日期**：2026-03-05
**涉及页面**：首页 / Explore / Gallery / Seedance
**问题等级**：高（用户体验核心路径）

---

## 一、问题描述

### 1.1 Gallery / Explore / Seedance 页面
用户在列表页下滑后，点击卡片打开 Modal 详情，点 X 关闭后：
- **异常现象 A**：界面回到页面顶部（滚动位置丢失）
- **异常现象 B**：点击关闭按钮无反应，但网站标题随点击次数变化，需多次点击才能关闭

### 1.2 首页 Latest Content 区域
用户在首页 Latest Content 区域点击 SrefCard：
- **异常现象 A**：详情 Modal 加载缓慢（需数秒才能显示内容）
- **异常现象 B**：点击关闭后，卡顿 5-10 秒，然后跳转到首页顶部（滚动位置归零）

---

## 二、根因分析

### 根因 1：`setSearchParams` 清除 `location.state`（关键）

**影响范围**：Explore.js、GalleryList.js、SeedanceList.js

这是整个问题链条的最根本原因。

```
用户在列表页操作 → setSearchParams({ replace: true }) 触发
         ↓
location.state（含 fromList: true）被完全清除
         ↓
Modal 的 handleClose() 检测不到 fromList
         ↓
执行 navigate('/explore') 而非 navigate(-1)
         ↓
PUSH 导航 → ScrollToTop 触发 → 滚动到顶部
```

**原始问题代码**：
```js
setSearchParams(params, { replace: true });
// replace: true 会替换当前历史条目，但 React Router 的 setSearchParams
// 不会自动继承当前的 location.state，导致 state 被清空
```

**修复**：在所有三个列表页补传 `state: location.state`：
```js
// Explore.js / GalleryList.js / SeedanceList.js
setSearchParams(params, { replace: true, state: location.state });
```

---

### 根因 2：Modal 打开时父列表页 URL 同步 Effect 重复触发（制造幽灵历史条目）

**影响范围**：Explore.js、GalleryList.js、SeedanceList.js

**问题序列**：
1. 用户点击卡片 → navigate 到 `/explore/123`（PUSH）
2. `navigationType` 变为 `'PUSH'`
3. URL 同步 `useEffect` 检测到变化（依赖数组包含 `navigationType`）
4. 条件判断 `navigationType !== 'POP'` → 通过
5. 执行 `setSearchParams(params, { replace: true })` → REPLACE 导航
6. `navigationType` 变为 `'REPLACE'`
7. Effect 再次触发 → 再次 REPLACE → 循环

**后果**：浏览器历史记录中积累大量重复条目。用户点击关闭后，`navigate(-1)` 只弹出一个条目，但历史栈中还有大量 `/explore` 或 `/explore/123` 的副本，导致：
- 浏览器后退按钮需要点击多次才能真正返回
- Helmet 因每次历史条目变化而更新标题，造成"标题在变但界面不动"的假象

**修复**：在 URL 同步 Effect 头部增加子路由（Modal）激活检测：
```js
// Explore.js
if (location.pathname !== '/explore') return;

// GalleryList.js
if (location.pathname !== '/gallery') return;

// SeedanceList.js
if (location.pathname !== '/seedance') return;
```

当 Modal 路由激活时（`/explore/123`），父列表页的 pathname 已不是根路径，Effect 直接返回，不执行 `setSearchParams`，从源头杜绝幽灵历史条目的产生。

---

### 根因 3：`document.body.style.overflow = 'hidden'` 导致 Chrome 重置 scrollY

**影响范围**：SrefModal.js、GalleryModal.js、SeedanceModal.js

Modal 打开时锁定 body 滚动是常见做法，但在 Chrome 中：
```
设置 overflow: hidden → 布局重排 → Chrome 将 scrollY 重置为 0
移除 overflow: hidden → 布局恢复 → scrollY 已经是 0，无法恢复
```

**修复**：移除所有 Modal 中的 overflow 锁定代码。Modal 使用全屏 backdrop 阻止背景交互，无需锁定 body 滚动：
```js
// 修复后（三个 Modal 均如此）
useEffect(() => {
  // 不锁定 body scroll — modal 有全屏 backdrop 阻止背景交互
  // overflow:hidden 会导致 Chrome 在移除时重置 scrollY 到 0
}, []);
```

---

### 根因 4：ScrollToTop 在 REPLACE 导航时对列表页错误触发

**影响范围**：ScrollToTop.js

当 Modal 关闭后执行 `navigate(-1)`（POP 导航），列表页的 URL 同步 Effect 此时正确跳过了（因为 `navigationType === 'POP'`）。但在某些场景下，REPLACE 导航类型的 ScrollToTop 仍会对列表根路由触发滚顶。

**修复**：在 ScrollToTop 中增加三重保护：
```js
useEffect(() => {
  if (navigationType === 'POP') return;                      // 后退导航跳过
  if (/\/(gallery|explore|seedance)\/[^/]+/.test(pathname)) return;  // Modal 路由跳过
  const isListRoute = /^\/(gallery|explore|seedance)\/?$/.test(pathname);
  if (navigationType === 'REPLACE' && isListRoute) return;   // 列表页 URL 参数同步跳过
  window.scrollTo({ top: 0, left: 0, behavior: 'instant' });
}, [pathname, navigationType]);
```

---

### 根因 5：首页 SrefCard 点击后 Explore.js 不必要的 API 调用

**影响范围**：首页 → 打开 `/explore/:id` Modal

**问题结构**：
```
首页点击 SrefCard
    ↓ navigate('/explore/123', { state: { fromList: true } })
    ↓
嵌套路由渲染（App.js 中 /explore/:id 是 /explore 的子路由）
    ↓
Explore.js 父组件强制挂载，立即发起：
  ① srefAPI.getPopularTags(40)    ← 标签列表，用户不需要
  ② srefAPI.getPosts({ page:1 })  ← 24条数据，用户不需要
  ③ SrefModal: srefAPI.getById(id) ← 用户真正需要的
```

三个请求并发，服务器同时处理，③ 的响应被 ①② 抢占资源，导致 Modal 内容加载显著变慢。

**修复**：在 Explore.js 中检测当前是否在列表根路径，仅在列表激活时才发起数据请求：
```js
const isListActive = location.pathname === '/explore';

// tags 查询
const { data: tagsData } = useQuery('sref-tags', fetchTags, {
  staleTime: 5 * 60 * 1000,
  enabled: isListActive,   // ← 新增
});

// sref 列表无限查询
const { data, ... } = useInfiniteQuery(['sref-list', ...], fetchList, {
  keepPreviousData: true,
  staleTime: 30000,
  enabled: isListActive,   // ← 新增
});
```

**效果**：从首页打开 Modal 时，Explore.js 静默挂载不发任何请求，服务器资源完全给 `getById(id)` 使用，Modal 内容加载速度显著提升。从 `/explore` 列表页打开 Modal 时，`isListActive = false`（pathname 已是 `/explore/:id`），列表数据已在内存中无需重新加载，一切正常。

### 根因 6：首页返回时 5-10 秒卡顿（缺少 keepPreviousData）

**影响范围**：Home.js

首页 (`/`) 和 Explore 页 (`/explore`) 是不同路由。用户从首页点击卡片进入 Modal：
1. 首页路由卸载（Home.js unmount）
2. 首页 react-query 缓存标记为 "inactive"
3. 用户关闭 Modal → `navigate(-1)` → 回到首页
4. Home.js 重新挂载，react-query 重新发起请求
5. 请求完成前 `isLoading = true`，滚动恢复 Effect 等待
6. 等待时间即 API 响应时间 → 卡顿感

**修复**：为 Home.js 的 useInfiniteQuery 添加 `keepPreviousData: true`：
```js
useInfiniteQuery(
  ['homeSrefs', { sort: sortBy, tag: selectedTag, search: debouncedSearch }],
  fetchFn,
  {
    staleTime: APP_CONFIG.CACHE.POSTS_STALE_TIME * 2,
    keepPreviousData: true,   // ← 新增
    refetchOnWindowFocus: false,
  }
);
```

**原理**：`keepPreviousData: true` 在查询 key 不变的情况下，即使数据需要 refetch，也保持 `isLoading = false`（`status = 'success'`），使用缓存中的旧数据立即渲染。滚动恢复无需等待，即时触发。

---

### 根因 7：首页滚动位置保存（Home.js 特殊性）

**影响范围**：Home.js

`/gallery`、`/explore`、`/seedance` 使用嵌套路由，列表组件 **永不卸载**，Modal 只是叠加渲染，自然保持滚动位置。

但首页 (`/`) 与 Explore (`/explore`) 是 **平级路由**，从首页进入 Modal 时首页完全卸载，需要手动保存/恢复：

```js
const HOME_SCROLL_KEY = 'home_scrollY';

// 卸载时保存当前滚动位置
useEffect(() => {
  return () => {
    sessionStorage.setItem(HOME_SCROLL_KEY, String(Math.round(window.scrollY)));
  };
}, []);

// POP 导航返回后，等数据就绪再恢复（避免 DOM 高度不足时滚动失效）
useEffect(() => {
  if (navigationType !== 'POP') return;
  if (isLoading) return;
  const saved = sessionStorage.getItem(HOME_SCROLL_KEY);
  if (!saved) return;
  sessionStorage.removeItem(HOME_SCROLL_KEY);
  requestAnimationFrame(() => {
    window.scrollTo(0, parseInt(saved, 10));
  });
}, [navigationType, isLoading]);
```

## 五、经验总结

1. **`setSearchParams` 会清除 `location.state`**：使用时必须透传 `state: location.state`，否则 Modal 的 `fromList` 标记会丢失。

2. **嵌套路由父组件的 Effect 需要感知子路由激活状态**：父组件的副作用（如 URL 同步）在子路由激活时应当静默，否则会干扰子路由的历史记录。

3. **`overflow:hidden` 在 Chrome 中会重置 scrollY**：SPA 中锁定背景滚动应使用 `position:fixed + top:-scrollY` 方案，或使用 backdrop 视觉阻断代替 DOM 锁定。

4. **嵌套路由 vs 平级路由的滚动恢复策略不同**：嵌套路由（列表+Modal）列表永不卸载，天然保持滚动位置；平级路由（首页→Explore Modal）需要 sessionStorage 手动保存/恢复。

5. **`enabled` 控制 Query 激活时机**：在父路由组件中，通过 `enabled: isListActive` 延迟数据加载，可避免子路由场景下的无效 API 调用，显著改善性能。
