## 架构与文件变更

### 新建文件

#### `server/services/veoService.js`
Google Vertex AI Veo 3.1 视频生成服务。

**核心流程**：
```
POST .../models/veo-3.1-generate-001:generateVideo
  → Long-Running Operation (name: "projects/.../operations/xxx")
  → GET .../operations/xxx 轮询（10s 间隔，最多 10min）
  → done: true → response.generateVideoResponse.generatedSamples[0].video.uri (GCS URI)
  → GET storage.googleapis.com/storage/v1/b/{bucket}/o/{obj}?alt=media 下载
  → 保存到 uploads/generated/veo_{ts}_{rand}.mp4
```

**认证方式**（双模式，优先 JSON 字符串）：
```js
// 方式1（生产推荐）: GOOGLE_SERVICE_ACCOUNT_JSON 环境变量存完整 JSON 字符串
const auth = new GoogleAuth({
  credentials: JSON.parse(process.env.GOOGLE_SERVICE_ACCOUNT_JSON),
  scopes: ['https://www.googleapis.com/auth/cloud-platform'],
});

// 方式2（本地开发）: GOOGLE_APPLICATION_CREDENTIALS 指向 JSON 文件路径
// 不传 credentials 时 GoogleAuth 自动读取该环境变量
```

**压缩 JSON 为单行（PowerShell）**：
```powershell
Get-Content service-account.json | ConvertFrom-Json | ConvertTo-Json -Compress -Depth 10 | Set-Clipboard
```

**关键参数**：
```js
parameters: {
  aspectRatio: '16:9',
  durationSeconds: validDuration,  // 4 | 5 | 6 | 7 | 8（Veo 只支持整数秒）
  addWatermark: false,
  generateAudio: Boolean(generateAudio),
  resolution: '720p' | '1080p',    // 不支持 480p
}
```

    if (m?.provider === 'dashscope') {

#### `server/routes/generate.js`

**Wan2.7 Image 处理器（关键坑点）**：

❌ **错误写法（导致 AccessDenied）**：
```js
// WRONG: multimodal-generation 不支持异步调用
headers: { 'X-DashScope-Async': 'enable' }
endpoint: 'text2image/image-synthesis'   // 也是错的
body: { input: { prompt } }              // 格式错误
parameters: { size: '1024*1024' }        // 分隔符错误
```

✅ **正确写法**：
```js
// 同步调用，直接得到结果（无需 task_id / 轮询）
endpoint: 'https://dashscope.aliyuncs.com/api/v1/services/aigc/multimodal-generation/generation'
headers: { 'Authorization': `Bearer ${DSKEY}`, 'Content-Type': 'application/json' }
// 无 X-DashScope-Async — multimodal-generation 是同步 API
body: {
  model: 'wan2.7-image',
  input: { messages: [{ role: 'user', content: [{ text: prompt }] }] },
  parameters: { size: '1024x1024', n: 1 }  // 用 x 不用 *，范围 768-2048px
}
signal: AbortSignal.timeout(120000)  // 2分钟（生成约需30-60s）

// 响应解析
const imgUrl = data.output?.choices?.[0]?.message?.content?.[0]?.image
            || data.output?.results?.[0]?.url  // fallback
```

**关键差异总结**：

| 项目 | 错误 | 正确 |
|------|------|------|
| 端点 | `text2image/image-synthesis` | `multimodal-generation/generation` |
| 调用模式 | 异步（`X-DashScope-Async: enable`） | **同步**（无该 header） |
| 请求体 | `input: { prompt }` | `input: { messages: [...] }` |
| 尺寸格式 | `"1024*1024"` | `"1024x1024"` |
| 响应解析 | `output.task_id` → 轮询 | 直接读 `output.choices[0]` |

**Size Map**：
```js
{ '1:1': '1024x1024', '4:3': '1024x768', '3:4': '768x1024', '16:9': '1365x768' }
```

**视频路由新增**：

  用于 `wan2-7-videoedit` 模式上传原始视频

## 已知问题与修复记录

### Bug 1: Wan2.7 图片显示不可用
**症状**: 模型列表中 Wan 2.7 图片不出现  
**根因**: `available: () => !!process.env.DASHSCOPE_API_KEY` — 服务器启动时 key 未被读取  
**修复**: 改为 `available: () => true`，在生成时检查 key

### Bug 2: `resultVideoUrl` 命名冲突
**症状**: 视频路由中 `const { videoUrl, taskId }` 的 `videoUrl` 覆盖请求体的 `videoUrl` 变量  
**修复**: 解构时重命名 `const { videoUrl: resultVideoUrl, taskId }`

### Bug 3: Wan2.7 Image `AccessDenied: current user api does not support asynchronous calls`
**症状**: 调用生图接口报 AccessDenied  
**根因**: `multimodal-generation/generation` 端点**不支持**异步调用（`X-DashScope-Async: enable`），该端点是同步 API  
**修复**:
  1. 移除 `X-DashScope-Async` header
  2. 直接解析同步响应 `output.choices[0].message.content[0].image`
  3. 超时由 30s 提高到 120s（生成约需 30-60s）

### Bug 4: Wan2.7 Image 端点和格式均错误（`url error`）
**症状**: `code: InvalidParameter, message: url error, please check url!`  
**根因**: 使用了错误端点 `text2image/image-synthesis`，以及错误的请求体格式  
**修复**:
  - 端点改为 `multimodal-generation/generation`
  - 请求体改为 messages 格式
  - 端点改为 `multimodal-generation/generation`（之前端点错才导致 `url error`，`*` 格式本身是对的）

### Bug 5: Veo 3.1 音频被错误禁用
**症状**: Veo 3.1 不显示音频选项  
**根因**: 误判文档说明，将 `generateAudio` 硬编码为 `false`  
**修复**: 恢复 `generateAudio: Boolean(generateAudio)`，并在前端恢复音频 toggle + 差价显示

### Bug 6: Wan2.7 r2v / videoedit 不可点击
**症状**: 两个子模式显示为灰色不可点  
**根因**: `WAN_SUB_MODES` 中两者 `comingSoon: true`  
**修复**: 改为 `comingSoon: false`，并新增 VideoZone 组件供 videoedit 上传视频

### Bug 7: Wan2.7 视频积分定价导致亏损
**症状**: 使用固定积分（25/30/40cr 每次），按实际 API 价格计算亏损  
**根因**: 初始方案使用固定 flat 积分，未考虑 DashScope 按秒计费  
**修复**: 改为按秒计费，720p=13cr/s，1080p=22cr/s（基于 ¥0.6/s 和 ¥1/s 官方价）

### Bug 8: Wan2.7 图片 `Invalid size format`
**症状**: `Invalid size format: 768x1024, expected format: width*height`  
**根因**: 将 size 分隔符从 `*` 改为 `x` 是错误的。原始的 `url error` 是由于端点错误引起，`*` 格式本身正确  
**修复**: sizeMap 改回 `*` 分隔符：`{ '1:1': '1024*1024', '4:3': '1024*768', '3:4': '768*1024', '16:9': '1365*768' }`

### Bug 9: Wan2.7 视频 resolution 格式错误
**症状**: `Input should be '1080P' or '720P': parameters.resolution`  
**根因**: videoService.js 中 sizeMap 使用像素尺寸字符串（`'1280*720'`），实际 API 要求大写字母标识（`'720P'`）  
**修复**: `sizeMap = { '720p': '720P', '1080p': '1080P', '480p': '480P' }`

### Bug 10: generate-history 图片无法显示（CORP 跨域阻断）
**症状**: 历史记录中所有图片显示为 alt 文字，网络层报 `ERR_BLOCKED_BY_RESPONSE.NotSameOrigin`  
**根因**: 两个叠加问题：
  1. `SERVER_PUBLIC_URL=https://iii.pics` 导致开发环境生成的图片 URL 指向生产域名，文件在本地不存在
  2. 旧记录 URL 为 `http://localhost:5500/uploads/...`，从 `localhost:3100` 跨端口加载被 CORP 阻断  
**修复**:
  - `server/index.js`：`/uploads` 静态文件加 `Cross-Origin-Resource-Policy: cross-origin`
  - `GenerationCard.js` `_resolveUrl`：**开发环境**将 `/uploads/` 绝对 URL 转为相对路径，走 CRA proxy（`package.json "proxy": "http://localhost:5500"`）；**生产环境**原样保留绝对 URL 不变

### Bug 11: Veo 3.1 创建请求 403（Vertex AI API 未启用）
**症状**: `Veo create failed [403]: Vertex AI API has not been used in project ... before or it is disabled`  
**根因**: Google Cloud 项目中 Vertex AI API 默认不启用  
**修复**: 在 GCP Console → APIs & Services → Vertex AI API → Enable（无需改代码）

### Bug 12: Veo 3.1 创建请求 404（错误 API 版本 + 错误方法名）
**症状**: `Veo create failed [404]` — HTML Google 404 响应页  
**根因**: 原始代码使用了错误的 API 路径和方法：
```
❌ v1beta1 + :generateVideo
   https://{loc}-aiplatform.googleapis.com/v1beta1/projects/.../models/veo-3.1-generate-001:generateVideo
```
Vertex AI Veo 的生产 API 实际为：
```
✅ v1 + :predictLongRunning
   https://{loc}-aiplatform.googleapis.com/v1/projects/.../publishers/google/models/veo-3.1-generate-001:predictLongRunning
```
**关键差异**：
- API 版本：`v1beta1` → `v1`
- 方法名：`:generateVideo` → `:predictLongRunning`
- 路径中需要 `publishers/google/`
- 请求体：`{ instances: [...], parameters: { sampleCount: 1, ... } }`（不是 `generateVideoRequest`）

**修复**（`veoService.js`）：
```js
const endpoint = [
  `https://${location}-aiplatform.googleapis.com/v1`,
  `projects/${projectId}/locations/${location}`,
  `publishers/google/models/${veoModel.apiId}:predictLongRunning`,
].join('/');

const requestBody = {
  instances:  [instance],
  parameters: { ...parameters, sampleCount: 1 },
};
```

### Bug 13: Veo 轮询 400（Operation ID must be a Long）
**症状**: `[veoService] poll 400: The Operation ID must be a Long. Received: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`  
**根因**: `predictLongRunning` 返回的是 UUID-based operation name（如 `projects/.../operations/abc123-uuid`），**不能**用标准 `/v1/.../operations/{id}` GET 接口轮询 — 该接口要求数字型 Long ID。  
**正确轮询方式**: 使用 `:fetchPredictOperation` POST 接口（官方文档 `projects.locations.publishers.models/fetchPredictOperation`）：
```
❌ GET  https://{loc}-aiplatform.googleapis.com/v1/{opName}
✅ POST https://{loc}-aiplatform.googleapis.com/v1/.../models/{modelId}:fetchPredictOperation
        body: { "operationName": "{opName}" }
```

**修复**（`veoService.js`）：
```js
// 与创建端点同一 models 路径，不同方法后缀
const fetchEndpoint = [
  `https://${location}-aiplatform.googleapis.com/v1`,
  `projects/${projectId}/locations/${location}`,
  `publishers/google/models/${veoModel.apiId}:fetchPredictOperation`,
].join('/');

// 轮询：POST + operationName 在 body 中
pollRes = await fetch(fetchEndpoint, {
  method:  'POST',
  headers: { 'Authorization': `Bearer ${freshToken}`, 'Content-Type': 'application/json' },
  body:    JSON.stringify({ operationName: opName }),
  signal:  AbortSignal.timeout(30000),
});
```

> 每次轮询前刷新 access token，因为生成最长需要 10min，而 Google OAuth token 有效期 1h（虽然不会到期，但仍做预防性刷新）

### Bug 14: Veo `done=true` 但视频 URI 为空（响应结构误判）
**症状**: 轮询返回 `done: true` 但报错 `Veo succeeded but no video URI`  
**根因**: 原代码试图读取 `data.response.generateVideoResponse.generatedSamples[0].video.uri`（GCS URI），但 `fetchPredictOperation` 实际响应结构完全不同：
```json
{
  "done": true,
  "response": {
    "videos": [
      { "bytesBase64Encoded": "AAAAAAAA..." }
    ]
  }
}
```
视频**内嵌于响应体**为 base64 编码，无 GCS URI。

**修复**（`veoService.js`）：
```js
// 原: const gcsUri = data.response?.generateVideoResponse?.generatedSamples?.[0]?.video?.uri;
// 新:
const videoB64 = data.response?.videos?.[0]?.bytesBase64Encoded;
if (!videoB64) throw new Error(`Veo succeeded but no video data. Response keys: ${Object.keys(data.response ?? {}).join(', ')}`);

// base64 解码并保存
const genDir   = path.join(__dirname, '../../uploads/generated');
fs.mkdirSync(genDir, { recursive: true });
const filename = `veo_${Date.now()}_${Math.random().toString(36).slice(2, 7)}.mp4`;
const filePath = path.join(genDir, filename);
fs.writeFileSync(filePath, Buffer.from(videoB64, 'base64'));
```

> 与 Wan2.7 video（返回临时 HTTP URL）不同，Veo 视频**直接 base64 内嵌**于 API 响应中，约 8-15MB。

### Bug 15: Veo 视频在浏览器中时长显示 0:00（moov atom 位置问题）
**症状**: 文件下载正常（8.7MB），但 `<video>` 元素时长为 0，无法 seek  
**根因**: MP4 文件的 `moov` 元数据原子（包含时长/帧率等信息）位于文件末尾（标准编码默认行为）。浏览器流式播放时需要先读到 moov 才能显示时长，末尾 moov 导致必须下载完整文件才能播放，部分浏览器显示 0:00。

**修复**: 使用 `ffmpeg -movflags +faststart` 将 moov atom 移到文件头部（"fast start"）：
```js
const fastFile = filePath.replace('.mp4', '_fs.mp4');
await new Promise((resolve) => {
  execFile('ffmpeg', ['-i', filePath, '-c', 'copy', '-movflags', '+faststart', '-y', fastFile],
    (err) => {
      if (err) {
        console.warn('[veoService] ffmpeg faststart failed, using raw file:', err.message);
      } else {
        fs.renameSync(fastFile, filePath); // 用 faststart 版本替换原文件
        console.log('[veoService] faststart applied');
      }
      resolve(); // 即使 ffmpeg 失败也继续（使用原文件）
    }
  );
});
```

> ffmpeg 命令：`-c copy` 仅重新封装不重新编码（极快），`-movflags +faststart` 移动 moov atom 到头部，`-y` 覆盖输出文件。

### Bug 16: Veo 视频在 generate-history 中无法播放（_resolveUrl 未应用到 videoUrl）
**症状**: generate-history 页面 Veo 视频卡片显示空白视频（无法播放），图片卡片正常  
**根因**: `GenerationCard.js` 的 `_resolveUrl` 函数（将绝对 URL 转换为 CRA proxy 可识别的相对路径）原本只应用于 `imageUrl`，未应用于 `videoUrl`  
**修复**（`GenerationCard.js`）：
```js
// 应用到 imageUrl 和 videoUrl 两者
const imageUrl = _resolveUrl(job.result?.imageUrl || job.imageUrl);
const videoUrl = _resolveUrl(job.result?.videoUrl || job.videoUrl);
```

`_resolveUrl` 函数（仅在开发环境生效，生产环境原样返回）：
```js
function _resolveUrl(url) {
  if (!url) return url;
  if (process.env.NODE_ENV !== 'development') return url;  // 生产环境不改变任何 URL
  // 把 http(s)://任意域名/uploads/ 转成 /uploads/ 走 CRA proxy
  return url.replace(/^https?:\/\/[^/]+(?=\/uploads\/)/, '');
}
```

## API 对比速查（最终正确版）

| 模型 | 供应商 | 端点 | 调用模式 | 结果获取 |
|------|--------|------|---------|---------|
| wan2.7-image | DashScope | `multimodal-generation/generation` | **同步** | 直接从响应体读取 `output.choices[0]` |
| wan2.7-t2v/i2v/r2v | DashScope | `video-generation/video-synthesis` | 异步（task） | 轮询 `GET /tasks/{id}` → `output.video_url` |
| wan2.7-videoedit | DashScope | `video-generation/video-synthesis` | 异步（task） | 同上 |
| veo-3.1 / fast / lite | Vertex AI | `v1/.../models/{id}:predictLongRunning` | LRO（长时操作） | POST `:fetchPredictOperation` 轮询 → `response.videos[0].bytesBase64Encoded` |

**Veo 完整调用链**：
```
1. POST .../models/{veo-model-id}:predictLongRunning
   body: { instances: [{ prompt, image? }], parameters: { aspectRatio, durationSeconds, generateAudio, resolution, addWatermark, sampleCount } }
   → { name: "projects/.../operations/uuid-xxx" }

2. POST .../models/{veo-model-id}:fetchPredictOperation  (每 10s 一次)
   body: { operationName: "projects/.../operations/uuid-xxx" }
   → { done: false }  → 继续等待
   → { done: true, response: { videos: [{ bytesBase64Encoded: "..." }] } }

3. Buffer.from(bytesBase64Encoded, 'base64') → 写入 uploads/generated/veo_{ts}_{rand}.mp4

4. ffmpeg -i {file} -c copy -movflags +faststart -y {file_fs} → 替换原文件（修复 0:00 时长）

5. 返回 ${SERVER_PUBLIC_URL}/uploads/generated/veo_{ts}_{rand}.mp4
```
